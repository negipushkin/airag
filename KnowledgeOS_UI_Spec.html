1. Design Intent

What this UI must communicate to the user at a glance

Primary job

The user asks a question in natural language and gets a grounded, cited answer. Every visual decision serves this one job. Nothing competes with it.

This is a professional tool for compliance officers, legal analysts, and enterprise PMs — not a consumer chatbot. The aesthetic should feel like a well-built internal tool: clean, information-dense, trustworthy. Zero decorative chrome.

The two things that differentiate this UI from a generic chat interface are:

  • Citation transparency — every answer surfaces exactly which document chunks were used, with source, page, section, and rerank score. The user should never have to wonder where an answer came from.
  • Pipeline visibility — the retrieval pipeline (decompose → embed → retrieve → rerank → synthesise) is surfaced as a trace bar on every answer. This is a trust signal, not just a debug tool.

4. Component Specifications

Implementation details for each component Claude Code needs to build or replace

DocumentSidebar Required
  • Props: documents[], activeDocId, onUpload, onSelectDoc, corpusStats
  • Document list item: icon, name (ellipsis), size+date, chunk count badge
  • Active state: white bg + 0.5px border
  • Stats grid: 2×2, live from props
  • Upload button: dashed border, hover fills white
AnswerCard Required
  • Props: answer, citations[], pipelineTrace, ragasScores
  • Answer text with inline CitationPill components
  • CitationPanel below answer text
  • PipelineTrace bar below citations
  • RAGASBar at bottom (conditional on scores)
CitationPanel Required
  • Props: chunks[], validationStatus
  • Header: source count + rerank info + validation badge
  • SourceCard per chunk: num, filename, score, section/page, excerpt
  • Click to expand full chunk text
  • Score badge: green if ≥ 0.80, amber if < 0.80
PipelineTrace Required
  • Props: stages[], totalMs
  • Stages: Decomposed, Embedded, Hybrid top-20, Reranked top-N, Synthesised
  • Each stage: small green pill when done, gray when pending
  • Total latency right-aligned
  • Do not omit — critical trust signal
QueryBar Required
  • Props: onSubmit, disabled, filterState
  • Auto-growing textarea, max 3 rows
  • Enter submits, Shift+Enter newline
  • Paperclip (attach), clock (history) icon buttons inside input
  • Send button: green, arrow-up icon
FilterChips Optional v1
  • Props: filters[], active, onChange
  • Chips: All docs, Policy, Contracts, SOPs, 2026 only
  • Maps to Pinecone docType + uploadedAt metadata filter
  • Active: green fill + border
  • Can be added after core Q&A works
RAGASBar Optional v1
  • Props: faithfulness, relevance, ctxPrecision
  • Only renders when scores are present
  • Green dot + value if ≥ threshold, amber if below
  • Thresholds: faithfulness 0.95, relevance 0.85, precision 0.80
EmptyState Required
  • Shown when no messages yet
  • Large muted icon, heading, subtext
  • 3 suggestion query buttons below
  • Suggestions: "What does our data retention policy say?", "Who approves access to PHI?", "Summarise the vendor contract obligations."

6. UI States

Empty / No messages
  • EmptyState component fills chat area
  • Muted icon (book or sparkles)
  • 3 suggestion query buttons
  • Filter chips still visible
Query loading
  • User bubble appended immediately
  • Assistant row with spinning loader
  • "Retrieving and re-ranking…" text
  • Send button disabled
  • Input disabled during generation
Streaming answer
  • Answer text streams token by token
  • Citation panel appears after stream ends
  • Pipeline trace renders after stream ends
  • Blinking cursor at stream end
Insufficient context
  • Amber-bordered answer card
  • Text: "I cannot find sufficient information…"
  • No citation panel shown
  • Pipeline trace still shown
  • Suggestion to refine query
Document uploading
  • Upload button replaced by progress bar
  • Toolbar badge → "Indexing… (N chunks)"
  • Doc appears in list with spinner
  • Chunk count fills in when done
API error
  • Red-bordered answer card
  • Error message + error code
  • Retry button in card
  • Rate limit shown distinctly

8. TDD Cross-References

Every UI element maps to a backend specification. Claude Code should wire these up, not hardcode them.

Citation panel header text — "top-20 retrieved → reranked to top-N" — N comes from pipelineTrace.rerankTopN. See TDD §5a.
Rerank score badges — scores come from Cohere Rerank v3 response. See TDD §5b. Green threshold = 0.80.
Citation validation badge — validationStatus: 'clean' | 'uncited_claims' | 'rerun' from API response. Show green shield on 'clean', amber warning on 'uncited_claims'. See TDD §6c.
Pipeline trace stages — driven by pipelineTrace.stages[] in API response. Each stage has name and durationMs. See TDD §3 data flow.
RAGAS scores — from ragasOnline field in API response. May be null if query was not sampled. Hide RAGASBar when null. See TDD §4.
Corpus stats — documents count, chunks count from vector store metadata. Faithfulness + latency from observability logger rolling average. See TDD §6.
Filter chips — map to Pinecone filter param: Policy → {docType: 'policy'}, Contracts → {docType: 'contract'}, 2026 only → {uploadedAt: {$gte: '2026-01-01'}}. See TDD §5a.
Insufficient context response — detected when answer starts with INSUFFICIENT_CONTEXT: prefix from synthesiser. Render amber card variant. See TDD §6b rule 2.

9. What Not to Do

Do not carry over from existing UI

The current UI is being replaced entirely. Do not preserve any existing layout, colour scheme, component structure, or interaction patterns unless they happen to match this spec. Start fresh from this document.

  • No generic chat bubble UI — this is not a chatbot. The answer card with citation panel is the core unit, not a plain text bubble.
  • No fabricated citation data — citation scores, filenames, page numbers must come from the API response. Never hardcode or mock them in the final build.
  • No omitting the pipeline trace bar — this is a differentiator, not a debug feature. It must appear on every successful answer.
  • No removing the RAGAS bar — render it conditionally (when scores are present), but do not delete the component.
  • No full-page loading spinners — loading states are localised to the assistant message row. The sidebar and filter chips remain interactive during query processing.
  • No hardcoded document list — the sidebar must be driven by the actual indexed document state from the vector store.
